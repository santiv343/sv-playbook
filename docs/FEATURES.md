# Features — what sv-playbook does and the problem each part solves

A living registry. Every capability exists because a real problem (often a real incident)
demanded it. Append a row when a feature lands; cite the incident/packet that birthed it.
The through-line: **take work and trust away from the agent — the CLI enforces, it never
relies on the agent remembering or being honest.**

> Long-term: this file should be GENERATED by the CLI from the done packets + events (each
> done packet is a feature + the problem it solved). Until that command exists, it is
> appended by hand as prose. (Prose docs are authored; operational state is CLI-only.)

## The board & lifecycle
| Capability | Problem it solves |
|---|---|
| Packets with a strict lifecycle (draft→ready→active→review→done, +blocked/dropped) | Agents wander; a fixed, legal-transition-only state machine bounds what can happen |
| Leases (30-min TTL, heartbeat, takeover) | Two agents (or a crash + retry) double-doing the same work |
| Write-set as declared blast radius + conflict check at `ready` | Two parallel workers editing the same files; scope collisions caught at planning time |
| CLI is the only author of packets | Hand-typed/hallucinated task files; the CLI validates every packet at write time |

## Anti-hallucination / anti-trust (the CLI verifies, never believes)
| Capability | Problem it solves |
|---|---|
| Evidence captured by the CLI on `active→review` (git HEAD sha + branch) | A worker fabricated a SHA; now the CLI reads it, the agent can't invent it (D24) |
| `[PLANNED]` write-set diff enforcement at review | Workers drift outside their blast radius and only the reviewer catches it (GATE-WRITESET-001) |
| `[PLANNED]` verify-green gate at review | "the agent says verify passed" — the CLI runs verify itself (GATE-VERIFY-001) |
| `[PLANNED]` evidence-required enforced at close | Packets closing with missing evidence (GATE-EVIDENCE-001) |
| `[PLANNED]` depends_on enforced at start | Starting a packet whose prerequisites aren't done (GATE-DEPS-001) |
| `[PLANNED]` CLI-only lint gate (no direct DB access) | Any agent (or the orchestrator) bypassing the CLI; a committed bypass can't merge (CLI-SOLE-INTERFACE-001) |

## Durability & recovery (never lose the board)
| Capability | Problem it solves |
|---|---|
| Real verified backups (VACUUM INTO snapshot, integrity + schema + sha256 checked restore) | A copyFileSync of a live DB could tear; a blind restore could clobber good data |
| Configurable backup dir outside `.svp/` | Losing `.svp/` took the backups with it |
| `[PLANNED]` unique backup filenames | Same-second backups collided and silently failed (FIX-BACKUP-FILENAME-001) |
| `[PLANNED]` `rebuild` from the committed git export + recovery-guard | A worker deleted `.svp` on a version mismatch; the CLI now never deletes and offers named recovery (STORE-RECOVERY-GUARD-001) |
| `[PLANNED]` migration safety (verified backup first, refuse with foreign leases) | A schema bump on a shared store corrupted concurrent workers (STORE-MIGRATION-SAFETY-001) |
| `[PLANNED]` shared store made concurrency-safe (WAL + transactions) | Keep one board for `serve` while staying safe under parallel workers (STORE-CONCURRENCY-001) |

## Roles, process & agent-agnosticism
| Capability | Problem it solves |
|---|---|
| EXEC/JUDGMENT role charters (planner/implementer/reviewer/orchestrator/product) | Dumb-model-proof procedures so a cheap model executes without ambiguity |
| Reviewer owns the merge; strict review; merge-gate branch protection | Nothing reaches main unreviewed; who-merges was contradictory across docs (sealed) |
| `[PLANNED]` merge→done mechanized + doctor drift flag | 9 packets stuck in review because the close step was manual (MERGE-CLOSE-001) |
| `[PLANNED]` instruction mirrors generated from one source | Non-Claude agents had no cold-start; agent-agnosticism (INSTRUCTIONS-MIRROR-001) |
| `[PLANNED]` `check <target>` deterministic validation of authored artifacts | The SDD-above layer had no mechanical gate (CHECK-001) |
| `[PLANNED]` worktree hygiene (WIP cap, hidden `.worktrees/`, auto-cleanup) | Workers left worktrees cluttering the workspace (WORKTREE-HYGIENE-001) |

## Doors & visibility
| Capability | Problem it solves |
|---|---|
| `[PLANNED]` `init` (new project) / `adopt` (existing project) | Two entry points: greenfield scaffold, and bringing a legacy repo under the playbook with a baseline |
| `[PLANNED]` `serve` (read-only live board) | The founder can't see what all the agents are doing in real time |
| `[PLANNED]` `handoff` (CLI-generated continuation prompt) | Relieving an agent without losing context; model relay (HANDOFF-CMD-001) |
| `status` / `doctor` | Re-entry after a break: board + health in two commands, zero cost |
